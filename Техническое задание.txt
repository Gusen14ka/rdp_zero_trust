Проект: Zero-Trust обёртка над RDP с поддержкой TCP и QUIC

1. Общая постановка задачи

Разработать систему защищённого удалённого доступа к рабочим столам, реализующую:

централизованный контроль доступа пользователей;
изоляцию целевых машин от внешней сети (Zero Trust);
туннелирование RDP-трафика через контролируемый сервер;
два независимых режима передачи данных:
TCP (базовый);
QUIC (экспериментальный);
экспериментальное сравнение эффективности TCP и QUIC в условиях деградации сети.

Система не реализует RDP-протокол, а использует существующие решения (например, xrdp) и строит над ними защищённый транспортный слой.

2. Архитектура системы

Система состоит из трёх основных компонентов:

2.1. Клиент

Приложение пользователя, которое:

проходит аутентификацию;
устанавливает соединение с сервером;
создаёт туннель;
проксирует RDP-трафик;
взаимодействует с локальным RDP-клиентом.
2.2. Центральный сервер (Control + Proxy)

Сервер объединяет:

2.2.1. Control Plane (TCP)

Отвечает за:

аутентификацию;
авторизацию;
выдачу сессий;
проверку прав доступа к машинам;
управление сертификатами;
контроль времени жизни сессий.

👉 Работает исключительно поверх TCP.

2.2.2. Data Plane (TCP или QUIC)

Отвечает за передачу данных:

TCP-режим:
классический прокси;
QUIC-режим:
передача RDP через QUIC-соединение.

👉 Это два полностью изолированных режима, без fallback.

2.3. Целевые машины
Linux/Windows хосты с RDP (например, xrdp);
полностью изолированы от внешней сети;
принимают подключения только от сервера.
3. Zero Trust модель

Система реализует следующие принципы:

отсутствует прямой доступ клиента к машине;
каждое подключение требует явной авторизации;
доступ выдаётся на конкретную машину;
доступ ограничен по времени;
все соединения аутентифицированы и зашифрованы;
используется собственный центр сертификации (CA).
4. Модель доступа
4.1. Пользователи

Каждый пользователь имеет:

уникальный идентификатор;
секрет (пароль / токен);
список доступных машин.
4.2. Машины

Каждая машина:

имеет уникальный ID;
зарегистрирована на сервере;
доступна только через прокси.
4.3. Политика доступа

Сервер хранит:

User → {Machine1, Machine2, ...}

Проверяется:

есть ли у пользователя доступ к машине;
не истёк ли срок сессии.
5. Система сертификатов (CA)
5.1. Требования

Реализуется собственный центр сертификации:

корневой сертификат;
сертификаты сервера;
(опционально) клиентские сертификаты.
5.2. Использование
Control Plane (TCP)
TLS для защиты канала управления;
QUIC
сертификаты используются в QUIC handshake (TLS 1.3 внутри QUIC).
5.3. Цели
исключить доверие к внешним CA;
обеспечить изоляцию;
реализовать Zero Trust.
6. Режимы работы системы
6.1. TCP-режим

Сценарий:

RDP client → local proxy → TCP → server → TCP → target machine

Особенности:

простой стрим байтов;
возможен head-of-line blocking;
базовая реализация.
6.2. QUIC-режим

Сценарий:

RDP client → local proxy → QUIC → server → TCP → target machine

Особенности:

QUIC поверх UDP;
встроенный TLS 1.3;
отсутствие head-of-line blocking между потоками;
устойчивость к потерям.
7. Потоки данных (важный момент)

В QUIC-режиме:

используется один основной поток для RDP (минимально необходимая реализация);

(опционально, если захочешь усилить работу)

можно добавить:
stream для control сообщений;
stream для telemetry.
8. Сценарий работы системы
8.1. Установление соединения
Клиент подключается к серверу (TCP, control plane);
Проходит аутентификацию;
Запрашивает доступ к машине;
Сервер:
проверяет права;
создаёт сессию;
выдаёт параметры подключения;
Клиент устанавливает Data Plane соединение:
TCP или QUIC;
Начинается проксирование RDP.
8.2. Работа сессии
сервер отслеживает TTL;
сервер может принудительно завершить сессию;
клиент получает сигнал завершения;
соединение закрывается.
8.3. Завершение

Причины:

истечение времени;
отзыв доступа;
ошибка соединения;
ручное завершение.
9. Требования к клиенту

Клиент должен:

поддерживать TCP и QUIC режимы;
работать как локальный прокси (например localhost:port);
проксировать RDP-трафик;
устанавливать control-соединение;
устанавливать data-соединение;
корректно завершать сессии.
10. Требования к серверу

Сервер должен:

поддерживать несколько пользователей;
поддерживать несколько машин;
хранить политики доступа;
реализовывать CA;
поддерживать TCP и QUIC транспорт;
управлять сессиями;
обеспечивать принудительное отключение.
11. Экспериментальная часть
11.1. Исследуемые решения
RDP
VNC
NoMachine
RDP-over-TCP (твоя реализация)
RDP-over-QUIC (твоя реализация)
11.2. Сетевые условия
нормальная сеть;
потери (например 2%);
высокий джиттер;
ограничение bandwidth;
комбинированные сценарии.
11.3. Метрики
latency (input lag);
стабильность изображения;
FPS / плавность;
реакция на потери;
время восстановления;
нагрузка CPU;
объём трафика.
11.4. Гипотеза

TCP страдает от head-of-line blocking при потерях пакетов,
QUIC должен обеспечивать более стабильную и отзывчивую работу.

13. Ключевые инженерные решения (важно для защиты)

Обязательно проговори это:

✅ 1. Разделение Control/Data plane
управление всегда TCP;
данные — TCP или QUIC.
✅ 2. Независимость безопасности от транспорта
security слой не меняется при переходе на QUIC.
✅ 3. Изоляция машин
нет прямого доступа;
только через сервер.
✅ 4. Два независимых режима
TCP ≠ fallback
QUIC — отдельный эксперимент.
✅ 5. Минимально достаточная реализация QUIC
один stream;
без overengineering.
14. Почему это выглядит сильно

У тебя получается не просто “прокси”, а:

Zero Trust система
с собственным CA
с разделением control/data plane
с экспериментом уровня сетевых протоколов
с реальным сравнением TCP vs QUIC

👉 Это уже уровень очень хорошей ВКР/курсовой.
